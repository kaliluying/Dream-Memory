# Dream Memory：跨项目、多 Agent 的分层共享记忆系统设计

## 1. 目标定义

### 一句话定位
把 Claude Code 和 Codex 的会话、工具结果与最终产物，提炼成可复用、可审核、可分层访问的共享记忆，供多个 Agent 在多个项目中持续使用。

### 它要解决的问题
1. 高价值信息只存在于聊天里，无法长期沉淀。
2. 多个 Agent 没有共享长期记忆，导致重复提问、重复判断。
3. 普通 RAG 只能找回原文，不能稳定形成可复用结论。
4. 原始会话噪音太大，不能直接作为共享上下文注入。

### 设计目标
- 支持跨项目。
- 支持多 Agent 共享。
- 支持分层可见。
- 支持候选记忆 -> 人工审核 -> 正式记忆。
- 支持底层原子事实 / 上层记忆卡片。
- 保持通用性，不绑死单一项目结构。

## 2. 核心原则

1. 梦境层只有提案权，没有正式入库权。
   - 任何正式共享记忆都必须人工审核。
2. 底层原子化，上层卡片化。
   - 底层便于去重、合并、冲突检测。
   - 上层便于 Agent 理解和使用。
3. 分层可见，而不是全量共享。
   - 记忆应按作用域组织，不同任务读不同层。
4. 证据优先。
   - 每条记忆都必须能追溯来源，而不是只留一句总结。
5. 宁可候选多，正式少。
   - 候选层可以宽松。
   - 正式层必须保守，避免污染共享记忆。

## 3. 系统分层

### 3.1 原始事件层
保存未加工材料，包括：
- Claude Code 会话
- Codex 会话
- 工具调用结果
- 最终产物
  - 计划
  - 文档
  - 代码变更摘要
  - 测试结果
  - 已写入的记忆文件

这一层的职责是保真，不是理解。

### 3.2 原子事实层
从事件中提取最小事实单元，例如：
- 用户偏好中文回答
- 当前系统需要分层可见模型
- 正式记忆必须人工审核
- 某项目的目标是 AI 项目研发助手

每条事实都带：
- 来源事件
- 时间
- 作用域
- 置信度
- 是否存在冲突

### 3.3 候选记忆层
把多个相关事实聚合成候选记忆卡片。
这层允许：
- 冗余
- 重复
- 多版本并存
- 低到中等置信度内容存在

它是梦境输出层。

### 3.4 审核队列层
候选记忆不会直接进入正式记忆，而是进入人工审核队列。
审核动作包括：
- 通过
- 拒绝
- 合并到已有记忆
- 修改后通过
- 标记需更多证据

### 3.5 正式记忆层
这里只放人工审核通过的正式共享记忆。
这是未来 Agent 的可信长期记忆来源。

### 3.6 Agent 注入层
Agent 启动或执行任务前，按任务动态读取相关记忆，而不是全量注入。

## 4. 记忆作用域设计

### 四层作用域
1. 全局层
   - 所有项目、所有 Agent 都可见。
   - 例如：用户长期协作偏好、全局工作方式。
2. 用户层
   - 与具体用户相关，但不一定适用于所有项目。
   - 例如：某类表达偏好、审阅偏好、沟通习惯。
3. 项目层
   - 只适用于某个项目。
   - 例如：该项目目标、外部依赖、长期约束、架构决议。
4. 会话层
   - 当前阶段临时有效。
   - 例如：本轮讨论达成的未审核结论、当前任务中的临时判断。

### 读取优先级建议
默认检索顺序：项目层 > 用户层 > 全局层 > 会话层补充。

更准确地说：
- 先按任务锚定项目。
- 再补用户偏好。
- 再补全局协作规则。
- 最后在当前会话层找短期上下文。

## 5. 核心对象模型

建议采用原子事实 + 聚合卡片混合模型。

### 5.1 原子事实
最小单位，适合机器处理。

建议字段：
- `id`
- `fact_type`
- `statement`
- `scope`
- `project`
- `session_id`
- `source`
- `evidence_refs`
- `confidence`
- `timestamp`
- `status`
- `conflicts_with`

示例：
- `statement: 用户偏好中文回答`
- `scope: global`
- `source: claude_code transcript`
- `confidence: 0.95`

### 5.2 候选记忆卡片
给梦境层输出的提案对象。

建议字段：
- `id`
- `title`
- `summary`
- `memory_type`
- `scope`
- `supporting_facts`
- `evidence_refs`
- `confidence`
- `stability`
- `review_status`
- `created_at`
- `updated_at`

### 5.3 正式记忆卡片
人工审核通过后的长期记忆对象。

在候选卡片基础上新增：
- `approved_by`
- `approved_at`
- `supersedes`
- `superseded_by`
- `visibility_rules`
- `retrieval_hints`

## 6. 梦境流水线

### 第 1 步：导入
从 Claude Code / Codex 收集：
- 对话
- 工具输出
- 产物

统一转成标准事件流。

### 第 2 步：归一化
把不同来源的材料统一成通用事件结构，例如：
- 谁说的
- 什么时候
- 属于哪个项目
- 是消息、工具结果还是产物
- 是否包含敏感信息

### 第 3 步：原子事实提取
从事件中提取最小可陈述事实。
这一层不直接做长期记忆判断，先只抽事实。

### 第 4 步：候选记忆生成
把高相关事实聚合成候选卡片，例如：
- 用户偏好类
- 项目背景类
- 决策结论类
- 参考资源类

### 第 5 步：去重与冲突检测
做 3 件事：
- 合并语义相近卡片
- 检测相互矛盾卡片
- 给出是更新旧记忆还是新增记忆的建议

### 第 6 步：进入人工审核队列
人工审核候选卡片：
- 通过
- 驳回
- 修改
- 合并
- 延迟决定

### 第 7 步：写入正式记忆
只有审核通过的内容，才能进入正式共享记忆层。

### 第 8 步：任务时动态注入
Agent 在执行任务前，基于：
- 当前项目
- 当前任务
- 时间新鲜度
- 记忆类型
- 可信度

来决定取哪些记忆。

## 7. 人工审核机制

### 审核原则
1. 梦境层不能直接写正式记忆。
2. 所有共享记忆都需人工确认。
3. 人工审核不只是过 / 不过。
   - 还要支持编辑、合并、延后。

### 审核时重点看什么
- 这条记忆是否真的会在未来复用？
- 它是否足够稳定，而不是临时说法？
- 是否有明确证据？
- 是否已经有近似正式记忆？
- 是否会误导其他项目或其他 Agent？

### 审核结果类型建议
- `approved`
- `rejected`
- `edited_and_approved`
- `merged`
- `needs_more_evidence`

## 8. Agent 如何使用这套记忆

### 使用方式
Agent 不应读完整个记忆库，而应做任务感知型取用。

### 建议的注入策略
对所有任务默认注入：
- 用户级偏好
- 全局协作规则
- 当前项目正式记忆摘要

对特定任务按需追加：
- 当前项目相关决策记忆
- 当前会话临时上下文
- 相关参考资源记忆

### 不应该默认注入的内容
- 未审核候选记忆全文
- 已冲突但未解决的记忆
- 与当前项目无关的项目级细节
- 太旧且长期未再被验证的内容

### 候选记忆怎么用
默认不直接注入主 Agent。
但可以在研究型 / 探索型 Agent 场景里，作为低置信补充线索。

## 9. 第一版范围建议

### V1 要做
1. 支持 Claude Code 与 Codex 两类来源。
2. 支持导入：
   - 对话
   - 工具结果
   - 最终产物
3. 建立标准事件模型。
4. 建立原子事实提取。
5. 生成候选记忆卡片。
6. 支持人工审核。
7. 审核通过后写入正式记忆。
8. Agent 读取正式记忆时支持分层过滤。

### V1 不做
1. 不做自动批准。
2. 不做完全自治的记忆合并。
3. 不做复杂向量检索优先架构。
4. 不做跨团队权限系统。
5. 不做强实时流式梦境更新。
6. 不做过于复杂的自我修正闭环。

## 10. 风险与难点

1. 候选记忆过多。
   - 如果提取太宽，会让审核队列爆炸。
2. 记忆污染。
   - 如果审核标准不严格，正式层会很快失真。
3. 冲突记忆处理困难。
   - 同一用户、同一项目在不同时期可能有不同结论。
4. 作用域判断不准。
   - 很多记忆到底是全局、用户级还是项目级，边界会模糊。
5. 证据链丢失。
   - 如果只保留总结，不保留来源，后续很难修正。

## 11. 最终结论

这不是一个聊天摘要器，而是一个跨项目、多 Agent、分层可见、人工审核驱动的长期共享记忆蒸馏系统。

其中：
- 原始事件层负责保真。
- 梦境层负责提炼候选记忆。
- 人工审核层负责守门。
- 正式记忆层负责长期共享。
- Agent 注入层负责按任务精确使用。
